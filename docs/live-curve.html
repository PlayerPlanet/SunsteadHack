What this proves

The agent that proposes can't apply, measure, or score — it reads the DB as a SELECT-only role over read-only MCP and returns a candidate; the substrate applies it and the frozen benchmark judges it. Every kept index passed a correctness check and a Welch's t-test against the noise floor. Round 3 looked plausible and was rejected anyway. That's the difference between a benchmark you can trust and a number an optimizer graded for itself.

Recipe: Postgres 16 + pg_stat_statements on :55432, seeded from proposer-container/seed.sql (scaled ~100×), workload job-prodyear = the title×cast_info production-year join. Each round: ClaudeCodeProposer().propose() → real CREATE INDEX → run_benchmark() (5 warmup / 15 timed, work_mem=256MB) → is_within_noise() → keep or rollback.
