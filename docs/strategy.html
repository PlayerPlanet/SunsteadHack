Coexistence — the neutrality fix

The only real conflict between hosting the benchmark and competing on it is neutrality. We dissolve it: BaaS ships private-by-default — a CI gate, not a public leaderboard. A customer benchmarks their own agent against their own bar; we never rank our house agents against theirs. Public boards stay opt-in, and our agents never enter them. Same substrate, two SKUs, no taste-test-hosted-by-the-brand problem.

Today — we ship BYO-task

Widen the bring-your-own surface one rung at a time. Every rung holds a demo.

BYO-task is real today because TaskRegistryStore already ingests JSON specs — task_id · objective · workload_id · action_space · db_ref · constraints. We don't open the floodgates; we widen each field of that spec in sequence, cashing a demoable win at each step. And because every run already executes inside the reversible membrane, even rung 01 is safe by construction.

01

Submit a task spec, see it runToday

Wire the dashboard's "new task" form straight into TaskRegistryStore. The schema already exists; we're exposing it. Action space stays a vetted catalog (shared_buffers, work_mem, index, statistics).

Demo: a teammate types a new task_id, picks levers, hits run — a fresh curve descends live on Aiven.

02

BYO objective + metric, from a menuToday

Let the submitter choose the oracle from a fixed menu — p99 latency / cost / join-plan quality. This sidesteps the universal-oracle trap: bounded choices, each with a judge we already trust.

Demo: two tasks, two different metrics, both produce a trustworthy, drift-checked curve side by side.

03

BYO workloadNext

Point workload_id / db_ref at a provided seed or snapshot, arm the drift tripwire against it. The benchmark now runs against data the customer brought.

Demo: a brought workload runs end-to-end; the stationarity tripwire flags — or clears — the result.

04

BYO agent over the wireNext

Register an external Proposer endpoint. The control plane drives it through run_loop exactly like a house agent — the leaderboard becomes meaningful.

Demo: a customer's agent and ours run the same task; both land on a private board with verified, replayable crossings.

05

Harden for the adversaryNext

Turn the cooperative boundary into a hostile-tenant boundary: action_space as a strict allowlist, reversibility via non-superuser PG perms, RunScopedLogClient isolation, max_iterations as a hard budget.

Demo: a deliberately hostile task tries an off-allowlist action and is blocked — the membrane holds, visibly.

The call

Why we go — and where we don't.

WHY WE SHOULD

  • The market's real pain is trust, not IQ. "Can I ship this agent?" beats "is it smart?" — and we measure trust on a live region.
  • Recurring by nature. Agents retrain constantly, so they re-benchmark constantly. A subscription, not a one-shot eval.
  • It leverages sunk cost. The multi-tenant rails, OAuth, and judge already exist. BYO-agent is a marginal add.
  • It owns a standard. Whoever hosts the gate owns the procurement decision in a task domain.

WHERE WE'RE CAREFUL

  • BYO-task is iterative, not a floodgate. Free-form oracles are a research program; we widen the spec field by field instead.
  • The threat model flips adversarial. Our boundary was built for cooperative agents — rung 05 exists precisely to harden it.
  • p99 needs no noisy neighbor. Trustworthy latency may mean dedicated runs; cost attribution is first-class.
  • Sell the decision, not the number. A gate in CI sells; a bare score doesn't.

BYO-agent yes. BYO-task today — widened one demoable rung at a time. Coexistence yes, because the benchmark stays a private gate, not a public ranking.
